## 2. 核心问题诊断

### 问题一：LLM 输出不可控，缺乏结构化

LLM 被要求一次性完成 6 个维度的对比 + Mermaid 图表，实际效果取决于模型能力。输出格式不稳定：有时产生有意义的对比，有时是泛泛而谈。没有 structured JSON schema 约束 LLM 输出格式。

### 问题二：数据被严重截断

`trunc = max(200, int(500 * 5 / max(N, 5)))` — 5 篇论文时每篇仅 500 字符，而各维度分析产物通常 3000–8000 字。LLM 对比依据极其有限。

### 问题三：可视化贫乏

只有 Mermaid 一种图表。D3.js v7 已在项目中用于趋势热力图和折线图，但未用于对比。缺少雷达图、对比热力图、桑基图、并列柱状图等。

### 问题四：无定量比较

四维度对比全部是定性文本。缺少自动提取的性能数值、各维度评分、论文相似度计算、公式/引用重叠度。

### 问题五：前端体验割裂

折叠卡片打开后是长文本，无法快速对比；Mermaid 在 iframe 中与主应用隔离无联动；论文颜色编码只在选择器使用，未贯穿到图表。

---

## 3. 改造目标

### 3.1 总体定位

从「一次性 LLM 文本 + Mermaid」升级为「**多层级、结构化、可交互的可视化对比分析工作台**」。

### 3.2 核心原则

1. **结构化优先** — LLM 输出 JSON Schema 约束，程序化提取对比数据
2. **渐进式对比** — 第一层概览 → 第二层维度表格 → 第三层 D3.js 图表
3. **D3.js 驱动** — 所有图表用 D3.js 在主应用内渲染，弃用 iframe
4. **颜色贯穿** — 论文颜色编码从选择器贯穿到所有图表和数据卡片
5. **数据可复用** — 对比结果结构化存储，支持历史对比列表和缓存复用
6. **离线可用** — 对比数据持久化到 JSON，不依赖 LLM 实时调用

### 3.3 目标体验对比

| | 当前 | 目标 |
|---|------|------|
| 概览 | 简单表格 | **对比仪表板**：论文信息卡 + 关键指标摘录 + 相似度热力图 |
| 维度对比 | 折叠 Markdown 卡片 | **并列对比表** (维度×论文矩阵) + **结构化条目提取** |
| 可视化 | Mermaid iframe | **5 种 D3.js 交互图** (雷达/热力/柱状/堆叠/桑基) |
| 定量分析 | 无 | **自动评分雷达图** + **指标提取排行榜** |
| 公式 | 无 | **公式相似度矩阵** + **共同符号统计** |
| 引用 | 无 | **引用重叠度** + **共引网络** |
| 导出 | Markdown | **Markdown** + **HTML 报告** + **CSV 数据表** |
| 历史 | 无 | **对比历史列表** (可重新查看/编辑/删除) |

## 4. 方案设计

### 4.1 后端架构改造

#### 4.1.1 新增结构化对比数据模型

```python
# models.py — 新增
@dataclass(slots=True)
class ComparisonDimension:
    """单个对比维度"""
    name: str                          # 维度名称
    paper_data: dict[str, str]         # paper_id → 原始文本
    extracted_scores: dict[str, float] # paper_id → LLM 评分 (1-10)
    extracted_items: dict[str, list]   # paper_id → [{name,description,type}]

@dataclass(slots=True)
class StructuredComparison:
    """结构化对比结果 (替代纯文本 synthesis)"""
    paper_ids: list[str]
    dimensions: dict[str, ComparisonDimension]
    metrics: list[dict]                 # [{paper_id, metrics: [{name,value,unit}]}]
    scores: dict[str, dict[str, float]] # paper_id → {维度: 评分}
    formula_overlap: dict[str, float]   # paper_pair → Jaccard 相似度
    similarity_matrix: list[list[float]] # N×N 余弦相似度矩阵
    chart_data: dict                    # 预计算图表数据 (JSON)
    synthesis_md: str                   # LLM 综合分析文本
    created_at: str
```

#### 4.1.2 5 阶段 LLM 对比流程

```
compare_papers() 改造为 5 阶段 Pipeline:

Phase 1: 结构化提取 (并行 asyncio.gather)
  └── 每篇论文 × 每维度 → LLM 提取结构化条目 + JSON Schema 约束
  └── 每篇论文 → LLM 提取性能指标 {name, value, unit, dataset, baseline}

Phase 2: 交叉评分 (并行)
  └── 每维度 → LLM 对每篇论文评分 (1-10) + 理由
  └── 产生产分矩阵 paper × dimension

Phase 3: 关联分析 (本地计算)
  ├── 公式重叠度 (Jaccard on LaTeX commands)
  ├── 引用重叠度 (共同引用计数)
  └── 论文相似度矩阵 (基于文本 embedding 余弦相似度)

Phase 4: 图表数据生成
  ├── 雷达图: scores 矩阵
  ├── 热力图: similarity 矩阵
  ├── 指标对比图: metrics 提取值
  └── 堆叠图: scores 按维度分解

Phase 5: 综合分析 (保留原 LLM 调用)
  └── 输入所有结构化数据 + 原文摘要 → LLM 写综合分析
```

```python
async def compare_papers_v2(self, paper_ids, provider, model, 
                             fast=False, dimensions=None):
    """v2: 结构化多阶段对比"""
    # ... 数据收集 (同原版 _compare_papers 前半部分) ...
    
    result = StructuredComparison(paper_ids=[...])
    
    if not fast and provider and model:
        # Phase 1: 结构化提取 (并行)
        result.dimensions = await self._structured_extract(papers, provider, model)
        
        # Phase 2: 交叉评分 (并行)
        result.scores = await self._cross_score(papers, result.dimensions, provider, model)
        
        # Phase 3: 关联分析 (本地计算)
        result.formula_overlap = self._compute_formula_overlap(papers)
        result.similarity_matrix = self._compute_similarity_matrix(papers)
        
        # Phase 4: 图表数据
        result.chart_data = self._build_chart_data(result)
        
        # Phase 5: 综合分析
        result.synthesis_md = await self._synthesize(result, provider, model)
    
    self._save_comparison_v2(result)
    return result
```

1. **向后兼容**: `/api/paper/compare` 响应新增 `structured` + `chart_data` 字段，原 `comparison_html` (Mermaid iframe) 保留为降级方案
2. **LLM 容错**: 结构化提取失败时降级到纯文本模式，不影响基本对比功能
3. **性能**: Phase 1-2 使用 `asyncio.gather` 并行 LLM 调用；Phase 3 纯本地计算 (< 1s)
4. **颜色系统**: `PAPER_COLORS` 扩展到 12 色，在图表和图例中保持一致
5. **响应式**: D3 图表使用 `viewBox` + 防抖 resize 重绘 (300ms)
6. **单文件约束**: 图表函数放在 `<script setup>` 中，不拆分组件文件
7. **LLM 模板**: 结构化提取必须使用 JSON Schema + `response_format` 或强制 JSON 输出

---

## 7. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| LLM JSON 输出格式不稳定 | JSON Schema 约束 + retry (1/2/4s) + 正则 fallback 提取 |
| 对比论文过多 (≥8 篇) | 热力图前 8 篇；雷达图前 6 篇；超出时警告并截断 |
| D3 图表面板过多一次渲染 | Tab 切换 + 惰性渲染 (仅当前可见 Tab 触发 render) |
| 后端计算耗时 | Phase 1-2 并行调用；缓存结果；提供 `fast` 跳过 LLM |
| 前端单文件 App.vue 过大 | 图表函数延迟定义 + 按需执行；未来可 Vite dynamic import |
| 论文分析未完成无法对比 | 优雅降级：跳过未完成论文 + 显示原因 |
