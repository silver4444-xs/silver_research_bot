# Silver Research Bot — 多维度完善分析报告

> 生成日期: 2026-06-15 | 基于项目源码全面审查

---

## 一、RAG 系统升级

### 当前状态
`research_rag.py` 使用**朴素 TF-IDF + 余弦相似度**，基于词频向量做检索。无分块策略优化、无语义理解、无图结构。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **GraphRAG** | 将文献内容抽取为实体-关系图（论文→作者→方法→数据集→指标），用图遍历替代纯关键词匹配。可引入 `NetworkX` + LLM 实体抽取，实现多跳推理式检索 |
| **向量化升级** | TF-IDF 词袋 → 接入 embedding 模型（OpenAI `text-embedding-3` 或本地 `sentence-transformers`），语义检索替换字面匹配 |
| **混合检索** | BM25 + 向量相似度 加权融合，互补精确术语匹配与语义意图 |
| **分块策略** | 当前整个文档入一个 chunk，应改为：Semantic Splitter（按 section 边界）+ 父子 chunk（小 chunk 检索，大 chunk 做上下文窗口） |
| **重排序** | 粗排→精排，检索 top-20 后接入 Cross-Encoder 重排保留 top-5 |
| **多模态 RAG** | 支持检索论文中的公式、图表、表格（当前 extractor 已提取但未索引） |
| **增量更新** | 当前入库需全量重写 JSONL，应支持增量 upsert + 删除 |

### 关键缺口
当前 RAG 完全没有利用 **extractor 已提取的结构化数据**（formulas/figures/tables），这些是最有价值的检索对象但被丢弃了。

## 二、Agent 记忆系统

### 当前状态
`memory.py` 的 `MemoryStore` 是纯 I/O 文件层：`MEMORY.md`（知识）、`history.jsonl`（历史）、`SOUL.md`（角色）、`USER.md`（用户画像）。`autocompact.py` 基于固定窗口（保留最后 8 条）做压缩。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **重要性评分记忆** | 每条记忆写入时附带 LLM 评估的重要性分数（1-10），压缩时优先保留高分记忆，低分记忆仅保留摘要。类似 MemGPT / Letta 的做法 |
| **记忆类型分层** | 当前所有记忆扁平。应分为：短期（最近 N 轮对话缓存）、中期（会话级压缩摘要）、长期（全局知识，如用户偏好、项目约定）三层，每层独立管理 |
| **遗忘曲线** | 基于 Ebbinghaus 遗忘曲线做记忆衰减：随时间递减的记忆权重，间隔重复触发"记忆巩固" |
| **冲突检测** | 新记忆写入时与已有记忆做语义相似度检测，发现矛盾（如"用户偏好 Python" vs "用户偏好 Go"）时标记并优先采信更新信息 |
| **主动记忆检索** | 当前记忆只在对话中被动引用。应改为：每轮对话前做记忆检索，将相关记忆注入上下文（`memory_retrieval` 工具） |
| **记忆迁移** | 跨会话记忆继承：同一项目不同 session 之间共享项目级记忆 |

### 关键缺口
`autocompact.py` 的压缩策略过于简单（纯计数窗口），没有对**记忆内容的语义重要性**做区分。`evaluator.py` 的通知过滤器是目前唯一用到"轻量 LLM 判断"的地方，这个模式可以推广到记忆评分。

## 三、MCP 与外部工具集成

### 当前状态
已有 MCP 支持基础框架：`agent/tools/mcp.py` 实现了 MCP server 集成，`config/schema.py` 定义了 MCP server 配置结构，`loop.py` 协调 MCP 工具注册。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **论文数据库 MCP** | 接入 arXiv / Semantic Scholar / PubMed / DBLP MCP servers，Agent 可直接搜索最新论文、获取 bibtex、跟踪引用关系 |
| **知识图谱 MCP** | 接入 Neo4j / NebulaGraph MCP，论文实体关系持久化到图数据库，支持 Cypher 查询 |
| **代码沙箱 MCP** | 已有 `sandbox.py`，可升级为隔离 Docker/K8s 沙箱 MCP，安全执行论文复现代码 |
| **Slack/飞书 MCP** | 将现有的 18 个 channel 实现包装为 MCP tools，让 Agent 可通过工具调用发送消息到任意渠道 |
| **浏览器自动化 MCP** | 接入 Playwright MCP，Agent 可自动打开论文网页、截图、提取交互式图表 |
| **文件管理 MCP** | 接入 Google Drive / OneDrive / Notion MCP，论文产物自动同步到云端 |
| **ArXiv RSS MCP** | 定时拉取 ArXiv 新论文，自动触发分析 pipeline |

### 关键缺口
当前 MCP 工具是**被动注册**的——工具存在但 Agent 不知道何时该用。缺少 **tool routing / tool recommendation** 层：根据用户意图自动选择最合适的 MCP 工具组合。

## 四、论文分析 Pipeline 增强

### 当前状态
6 阶段线性 Pipeline：Extract → Translate → 4 维分析 → Formula → Visualize → Audit。每个阶段依赖前一阶段产出。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **引用图谱** | 新增 Stage: 解析参考文献 → 检索被引论文 → 构建引用图谱（DAG），可视化论文的知识继承关系 |
| **复现能力** | 新增 Stage: 从论文中提取算法伪代码 → LLM 转可运行代码 → 沙箱执行 → 对比论文声称指标 |
| **多论文对比增强** | 当前 `compare` 仅做 LLM 横向对比。可加入: 方法谱系图（谁引用了谁的方法）、指标排行榜（相同数据集上不同方法的分数对比）、趋势时间线 |
| **增量分析** | 当同一篇论文的新版本（v2→v3）上传时，只分析 diff 部分，不重跑全 Pipeline |
| **A/B 审稿** | 模拟多个审稿人视角（理论家/工程派/领域专家），从不同侧重点审读论文，生成多份审稿意见 |
| **交互式分析** | 当前是 fire-and-forget。应支持用户在分析过程中提问："这个公式的物理含义是什么？"——Agent 在分析产物基础上做增量回答 |
| **音频输出** | 调用 TTS 将翻译/分析结果生成播客式音频摘要（已有 `transcription.py`） |

### 关键缺口
Pipeline 完全是**线性的**——如果某个阶段产出质量差，没有自动回退/修正机制。AUDIT 阶段发现问题后只会标记不会修复。应该加入 **self-correction loop**（审计发现问题 → 回到对应阶段重新生成）。

## 五、Agent 基础设施

### 当前状态
`AgentLoop`（`loop.py`）+ `AgentRunner`（`runner.py`）提供完整的 ReAct 运行时。`SubagentManager` 支持子 Agent 派生。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **多 Agent 协作** | 当前只有 spawn 子 Agent。可升级为: 论文分析专属 Agent 团队（翻译 Agent + 分析 Agent + 审计 Agent）通过 MessageBus 协作，类似 AutoGen/CrewAI 模式 |
| **Agent 角色工厂** | 基于 `SOUL.md` 模板化：论文审稿 Agent、代码审查 Agent、文献综述 Agent，每个有专属 tools 和 system prompt |
| **Human-in-the-loop** | 关键决策点（如审计不通过是否重试、分析方向选择）暂停并等待人工确认 |
| **流式中间结果** | 当前 6 阶段只透出进度条。应支持 WebSocket 推送每个阶段的中间产物流式输出 |
| **Agent eval 框架** | 无测试。应建立: 标准测试论文集 + 预期分析结果 + 自动评分（翻译 BLEU、分析覆盖度、公式准确率） |
| **成本控制** | 当前无 token 预算管理。应加入: 每篇论文分析预算上限、月度总预算、模型分层（便宜模型做提取，贵模型做分析） |

### 关键缺口
`cli/models.py` 的模型信息是**全空的 stub**（litellm 移除后未补回）。这意味着 `get_model_context_limit()` 返回 None，token 计数相关功能可能异常。

## 六、前端与交互体验

### 当前状态
Vue 3 单文件 SPA，深色科技风，3 标签页。论文分析结果通过静态渲染展示。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **论文阅读器** | 当前翻译/分析都是纯文本渲染。应加入: 双栏对照（原文 vs 译文）、公式高亮点击弹解释、图表画廊 |
| **Dashboard** | 首页仪表盘: 总分析论文数、公式数、各维度覆盖度雷达图、最近活动时间线 |
| **国际化** | 当前中文硬编码。应提取 i18n |
| **移动端适配** | 当前侧边栏 + 主区域布局在手机上崩溃。响应式改造 |
| **离线能力** | Service Worker + IndexedDB，缓存已分析论文结果 |
| **论文订阅** | 按关键词/作者/领域订阅新论文，自动拉取+分析 |
| **社交功能** | 论文评注分享、公开榜单（最受关注论文） |
| **交互式可视化** | 当前 Mermaid 是静态渲染。可升级为 D3.js 交互式图表（可缩放、可点击节点查看详情） |

## 七、可观测性

### 当前状态
`loguru` 日志（无结构化）+ `progress.json` 进度文件 + `research_app.py` 的 try/catch → HTTPException。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **结构化日志** | 统一 JSON 格式日志，含 trace_id / paper_id / stage / duration_ms |
| **OpenTelemetry 追踪** | 每个 LLM 调用、Pipeline 阶段、工具执行都生成 span，可导出到 Jaeger/Grafana |
| **LLM 调用监控** | 记录每次 LLM 调用的 model/tokens/prompt/response/latency，生成成本报告和延迟分布 |
| **告警** | Pipeline 失败 / LLM 调用超时 / 磁盘空间不足 → 通过现有 channel 推送通知 |
| **Dashboard** | 实时展示: 正在分析的论文、各阶段耗时分布、LLM 调用次数与成本 |

## 八、安全加固

### 当前状态
`security/network.py` 有 SSRF 白名单保护。`sandbox.py` 提供代码执行隔离。

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **输入验证** | PDF 上传无文件大小限制、无恶意文件检测（PDF 可嵌 JavaScript）。应加入: 文件大小限制、ClamAV 扫描、PDF 结构校验 |
| **LLM 输出过滤** | 无输出安全过滤。LLM 生成的 HTML（visualizer/formula）直接内嵌 iframe srcdoc。应做 XSS 过滤 |
| **API 限流** | 当前无任何 rate limiting。应加入: `/api/paper/upload` / `/api/agent/chat` 按 IP/API Key 限流 |
| **认证鉴权** | 当前完全无认证。应加入: API Key / JWT 认证，多用户隔离 |
| **敏感信息过滤** | 用户上传的论文内容经 LLM 发送到外部 API，应过滤可能的 PII/密钥 |

## 十、性能与规模

### 改进方向

| 方向 | 具体方案 |
|---|---|
| **PDF 解析并行** | 大量 PDF 批处理时，用 `ProcessPoolExecutor` 并行解析 |
| **LLM 结果缓存** | 相同 prompt + 相同论文 → 缓存结果（用 prompt + text 的 hash 做 key），避免重复调用 |
| **增量索引** | `index.json` 当前全量读写。论文数 > 10K 时应迁移至 SQLite |
| **前端构建优化** | 单文件 SPA 随功能增长会膨胀。考虑: 代码分割、懒加载分析组件 |
| **工作区清理** | 无自动清理。应加入: 定时清理超过 N 天未访问的临时文件、导出 ZIP 缓存 |

## 总结：改进优先级矩阵

按**投入产出比**排序：

| 优先级 | 维度 | 改进项 | 理由 |
|---|---|---|---|
| **P0** | 测试 | 核心模块单元测试 | 零测试是最大风险 |
| **P0** | 安全 | 文件大小限制 + 输出 XSS 过滤 | 直接可被利用 |
| **P1** | RAG | 向量化 + 混合检索 | 当前 TF-IDF 检索几乎不可用 |
| **P1** | 记忆 | 重要性评分记忆管理 | 长期使用场景的核心体验 |
| **P1** | Pipeline | 审计发现自动修复 loop | 提升分析质量闭环 |
| **P1** | MCP | arXiv/Semantic Scholar MCP | 论文场景最直接的价值增量 |
| **P2** | Agent | 多 Agent 论文审稿团队 | 差异化竞争力 |
| **P2** | 前端 | 双栏对照论文阅读器 | 核心使用体验提升 |
| **P2** | 可观测 | LLM 调用成本追踪 | 实际运营需要 |
| **P3** | GraphRAG | 实体关系图检索 | 工程量大，但效果上限高 |
| **P3** | 复现能力 | 算法→代码→执行→对比 | 技术难度高，但独特性强 |
